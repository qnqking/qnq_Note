### 🧠 详细核心笔记梳理

#### 1. Git 与 GitHub：版本控制的基石

- **Git**：是一个开源免费的版本控制系统。它像拍照一样记录仓库的每一个状态（快照），形成一条可回溯的历史链路。
- **GitHub**：是全球最大的代码仓库托管平台。它将全球开发者的Git仓库集合在一起（Hub），支持公开（开源）和私有仓库。
- **核心逻辑**：**仓库（Repository）** 是核心，分为本地仓库（Local）和远端仓库（Remote）。

#### 2. 核心操作单元：Commit（提交）

- **定义**：每一次保存历史版本的操作。
- **组成**：
    - **Commit ID**：由哈希算法生成的唯一标识（长ID和短ID）。
    - **Commit Message**：提交摘要，描述做了什么。
- **AI 时代技巧**：每次让AI完成一个小功能点就做一次Commit。如果AI改错了，可以直接通过自然语言让AI回退，无需记忆命令。

#### 3. 分支管理：Branch 与 Head

- **Branch（分支）**：
    - 默认主干是 `main` 或 `master`。
    - 创建分支（如 `feature`）是基于当前代码的指针，不占用额外空间。
    - 在分支上的修改不会影响主干，开发完后通过 **Merge（合并）** 回主干。
- **Head**：指向当前仓库处于哪一次Commit。
    - **分离头指针（Detached Head）**：直接检出（Check out）某次历史Commit的状态，不推荐在此状态下修改代码，容易丢失。
- **Worktree**：Git工作树，本质是创建一个新分支并完整复制到新文件夹，允许主文件夹和分支文件夹并行开发、互不干扰。

#### 4. Git 分区与文件流转

Git操作的本质是在不同分区之间同步文件：

1. **Working Directory（工作区）**：你电脑上的本地文件夹。
2. **Staging Area（暂存区）**：Commit前的检查点（Stage Changes）。现代工具（如VS Code）通常将 `add` 和 `commit` 合并为一步，简化了操作。
3. **Local Repository（本地仓库）**：提交（Commit）后保存在这里。
4. **Remote Repository（远端仓库）**：如GitHub。通过 `Push` 推送代码，通过 `Pull`（Fetch + Merge）拉取他人代码。

#### 5. GitHub 多人协作与开源贡献 (Fork & PR)

这是GitHub最核心的协作流程，分为两种场景：

- **场景一：开源贡献（无直接权限）**
    
    1. **Fork**：将别人的仓库复制一份到自己的名下。
    2. **Clone**：将自己名下的仓库克隆到本地。
    3. **Develop**：创建分支修改代码，Commit并Push到自己名下的远端仓库。
    4. **Pull Request (PR)**：在GitHub上发起请求，请求将你的分支合并进原作者的主干。
    5. **Code Review**：原作者审核代码，无误后点击Merge合并。
- **场景二：团队协作（有权限）**
    
    - 管理员将开发者添加为 **Collaborator（协作者）**。
    - 开发者无需Fork，直接Clone主仓库。
    - 在本地创建分支开发，Push到远端，然后直接发起PR合并回主干。

#### 6. 高频概念与“后悔药”

- **Merge Conflict（合并冲突）**：当两个分支修改了同一文件的同一行代码时发生。AI可以辅助解决冲突（如选择保留哪个版本或合并两者）。
- **.gitignore**：配置文件，声明哪些文件（如密钥、依赖包 `node_modules`）不应被Git管理。
- **三大“后悔药”**：
    - **Discard**：放弃**未Commit**的更改（仅限本地未提交）。
    - **Reset**：强制回退到某个历史状态（会丢弃之后的提交，禁止在多人协作分支上使用）。
    - **Revert**：生成一个反向提交来抵消某次Commit（安全操作，适合多人分支）。

#### 7. 进阶操作（了解即可，交由AI执行）

- **Cherry-pick**：拣选提交。不合并整个分支，只挑选特定的某几次提交合并进来。
- **Stash**：临时存储。代码写一半未写完，需要切换分支时，将未完成的代码暂存起来，回头再取出来继续写。
- **Rebase（变基）**：将一个分支的根基变更到另一个分支上。它会让历史线变得干净（不产生多余的Merge Commit），但会改写历史，通常需要强制推送（Force Push），**严禁在多人公用的分支上使用**。
